Authentication & Making Callsâ€‹

Making API calls with DataHub requires authentication, which can be performed by passing your API key via the request header Authorization or as a parameter in the query URL.

Here's some boilerplate to get you started.

  • Fetch with Header
  • Fetch with Parameter
  • CURL with Header
  • CURL with Parameter
fetch("https://polkadot-slate.datahub.figment.io/api/v1/flows", {
    method: 'POST', // can also be 'GET', 'PUT' or any appropriate method
    headers: {
        "Authorization" : "<api_key>",
        "Content-Type": "application/json" // if method is 'POST'
    },
    body: JSON.stringify({
      // JSON Payload
    })
})
fetch("https://polkadot-slate.datahub.figment.io/apikey/<api_key>/api/v1/flows", {
    method: 'POST', // can also be 'GET', 'PUT' or any appropriate method
    headers: {
        "Content-Type": "application/json" // if method is 'POST'
    },
    body: JSON.stringify({
      // JSON Payload
    })
})
curl -X POST 'https://polkadot-slate.datahub.figment.io/api/v1/flows' \
--header 'Content-Type: application/json' \
--header 'Authorization: <api_key>' \
--data '{
  // JSON Payload
}'
curl -X POST 'https://polkadot-slate.datahub.figment.io/apikey/<api_key>/api/v1/flows' \
--header 'Content-Type: application/json' \

  // JSON Payload

New Staking Flow#

Create a new Staking flow with a POST request to /api/v1/flows.

View specification

Parameters

  • flow - object
    • network_code - string Network this flow operates on (ex. polkadot) (required)
    • chain_code - string Chain this flow operates on (i.e., mainnet) (required)
    • operation string The operation to perform (i.e., staking) (required)
    • version - string API version to use (ex. v1) (optional)

Results

  • id - string ID of the newly created flow.
  • state - string The current state of the flow i.e., initialized.
  • actions - array The actions array contains the name of the next action in the flow and details of the expected inputs in the data field.

Assign Stash and Controller Account#

Use the assign_stash_and_controller_account action if you already have a controller account set up and wish to add bond to it.

View specification

Parameters

  • name - string Name of the action to execute (required).
  • inputs - array
    • controller_account_address - string Address of the wallet DOT will be bonded to (required).
    • controller_proxy_account_address - string A proxy account that will sign on behalf of the controller account (optional).
    • stash_account_address - string Primary wallet from which DOT will be bonded (required).
    • stash_proxy_account_address - string A proxy account that will sign on behalf of the controller account (optional).
    • validator_address - string Address of the validator to whom you want to delegate your tokens.
    • reward_destination - string Reward destination. (required) Indicates whether rewards are to be:
    1. Added to the staked balance (staked)
    2. Sent to the controller account, liquid (controller)
    3. Sent to the stash account, liquid (stash)
    • amount - number - The amount of DOT to be bonded to the controller. This field should be in full DOT, not in Planck (ie. if you want to stake 5 DOT, return "amount": 5.0). Staking API will translate the amount to Plancks so you do not have to worry about the conversion factor (required).

Results

  • id - string ID of the newly created flow.
  • state - string Current state of the flow.
  • actions - array The actions array contains the name of the next action in the flow and details of the expected inputs in the data field.

Create Bonding Transaction#

Use the create_bonding_tx action if you do not have a controller account set up and would like to create a controller and bond tokens.

View specification

Parameters

  • name - string Name of the action to execute (required).
  • inputs - object
    • controller_account_address - string Address of the wallet DOT will be bonded to (required).
    • controller_proxy_account_address - string A proxy account that will sign on behalf of the controller account (optional).
    • stash_account_address - string Primary wallet from which DOT will be bonded. Cannot have a zero balance. (required).
    • stash_proxy_account_address - string A proxy account that will sign on behalf of the controller account (optional).
    • reward_destination - string Reward destination. (required) Indicates whether rewards are to be:
    1. Added to the staked balance (Staked), note the uppercase, staked will not work.
    2. Sent to the controller account, liquid (Controller), note the uppercase, controller will not work.
    3. Sent to the stash account, liquid (Stash), note the uppercase, stash will not work.
    • amount - number - The amount of DOT to be bonded to the controller. Minimum of 1.0. This field should be in full DOT, not in Planck (ie. if you want to stake 5 DOT, return "amount": 5.0). Staking API will translate the amount to Plancks so you do not have to worry about the conversion factor (required).

Results

Note that in this response you can find the unsigned, serialized transaction in data[create_bonding_tx][raw].

  • id - string ID of the newly created flow.
  • state - string Current state of the flow.
  • actions - array The actions array contains the name of the next action in the flow and details of the expected inputs in the data field. There are two possible actions:
  • refresh_bonding_tx - If you need to get a fresh version of the transaction Payload (i.e., if the nonce needs to be updated).
  • sign_bonding_tx - Submit a signed transaction Payload ready to be broadcast to the network.

  • removed - The transaction has been confirmed and the proxy is revoked.

The Remove Proxy flow is now complete!

  "name": "sign_remove_proxy_tx",
  "inputs": {
    "transaction_payload": null,
